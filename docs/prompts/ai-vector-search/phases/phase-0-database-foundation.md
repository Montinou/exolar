## Overview

1. Enabling the pgvector extension in Neon
2. Adding the `error_embedding` vector column to `test_results`
3. Creating HNSW index for fast similarity search
4. Adding helper functions for vector operations

### Step 0.2: Add Vector Column to test_results

**Add to same migration file:**

```sql
-- ============================================
-- STEP 2: Add embedding column to test_results
-- ============================================
-- Using 768 dimensions for Gemini text-embedding-004
DO $$
BEGIN
  IF NOT EXISTS (
    SELECT 1 FROM information_schema.columns
    WHERE table_name = 'test_results'
    AND column_name = 'error_embedding'
  ) THEN
    ALTER TABLE test_results
    ADD COLUMN error_embedding vector(768);

    COMMENT ON COLUMN test_results.error_embedding IS
      'Vector embedding of error_message + stack_trace for similarity clustering. Generated by Gemini text-embedding-004.';
  END IF;
END $$;
```

**Why 768 dimensions?**
- Gemini `text-embedding-004` produces 768-dimensional vectors
- Matches the planned embedding model exactly
- Good balance of accuracy vs storage

- [ ] Comment explains column purpose

### Step 0.3: Create HNSW Index

**Add to same migration file:**

```sql
-- ============================================
-- STEP 3: Create HNSW index for fast similarity search
-- ============================================
-- HNSW (Hierarchical Navigable Small Worlds) provides:
-- - Sub-millisecond query times
-- - Better recall than IVFFlat
-- - No need for periodic retraining

-- Partial index: only index failed tests (where embeddings exist)
CREATE INDEX IF NOT EXISTS idx_test_results_error_embedding_hnsw
ON test_results
USING hnsw (error_embedding vector_cosine_ops)
WHERE error_embedding IS NOT NULL;

-- Additional partial index for faster filtering by execution
CREATE INDEX IF NOT EXISTS idx_test_results_execution_embedding
ON test_results (execution_id)
WHERE error_embedding IS NOT NULL;
```

**Index Strategy:**
- `vector_cosine_ops` for normalized embeddings (Gemini embeddings are normalized)
- Partial index saves space (only ~5% of tests fail)
- Secondary index enables fast filtering by execution_id

**Verification:**
- [ ] HNSW index created with cosine distance
- [ ] Partial index for space efficiency

### Step 0.4: Add Helper Functions

**Add to same migration file:**

```sql
-- ============================================
-- STEP 4: Helper functions for vector operations
-- ============================================

-- Function: Find similar failures within an execution
CREATE OR REPLACE FUNCTION find_similar_failures(
  p_execution_id INTEGER,
  p_embedding vector(768),
  p_threshold FLOAT DEFAULT 0.1,  -- cosine distance threshold
  p_limit INTEGER DEFAULT 10
)
RETURNS TABLE (
  id INTEGER,
  test_name TEXT,
  error_message TEXT,
  similarity FLOAT
)
LANGUAGE SQL
STABLE
AS $$
  SELECT
    tr.id,
    tr.test_name,
    tr.error_message,
    1 - (tr.error_embedding <=> p_embedding) as similarity
  FROM test_results tr
  WHERE tr.execution_id = p_execution_id
    AND tr.error_embedding IS NOT NULL
    AND tr.error_embedding <=> p_embedding < p_threshold
  ORDER BY tr.error_embedding <=> p_embedding
  LIMIT p_limit;
$$;

-- Function: Find similar failures across all executions (for historical analysis)
CREATE OR REPLACE FUNCTION find_similar_failures_global(
  p_organization_id INTEGER,
  p_embedding vector(768),
  p_threshold FLOAT DEFAULT 0.15,
  p_limit INTEGER DEFAULT 20
)
RETURNS TABLE (
  id INTEGER,
  execution_id INTEGER,
  test_name TEXT,
  error_message TEXT,
  similarity FLOAT,
  created_at TIMESTAMP WITH TIME ZONE
)
LANGUAGE SQL
STABLE
SECURITY DEFINER
AS $$
  SELECT
    tr.id,
    tr.execution_id,
    tr.test_name,
    tr.error_message,
    1 - (tr.error_embedding <=> p_embedding) as similarity,
    tr.created_at
  FROM test_results tr
  INNER JOIN test_executions te ON tr.execution_id = te.id
  WHERE te.organization_id = p_organization_id
    AND tr.error_embedding IS NOT NULL
    AND tr.error_embedding <=> p_embedding < p_threshold
  ORDER BY tr.error_embedding <=> p_embedding
  LIMIT p_limit;
$$;

COMMENT ON FUNCTION find_similar_failures IS
  'Find failures with similar error embeddings within a single execution. Used for smart clustering.';

COMMENT ON FUNCTION find_similar_failures_global IS
  'Find failures with similar error embeddings across all executions in an organization. Used for historical analysis.';
```

**Function Design:**
- `find_similar_failures` - For clustering within a single run
- `find_similar_failures_global` - For finding historical similar failures
- `SECURITY DEFINER` on global function to respect RLS

**Verification:**
- [ ] Both functions created
- [ ] Functions have appropriate comments
- [ ] Organization filtering included

### Step 0.5: Add Clustering Support Table (Optional)

**Add to same migration file:**

```sql
-- ============================================
-- STEP 5: Cluster cache table (optional optimization)
-- ============================================
-- Stores pre-computed clusters for faster dashboard loading
-- Can be populated lazily or via background job

CREATE TABLE IF NOT EXISTS failure_clusters (
  id SERIAL PRIMARY KEY,
  execution_id INTEGER NOT NULL REFERENCES test_executions(id) ON DELETE CASCADE,
  cluster_id INTEGER NOT NULL,  -- Cluster number within execution (1, 2, 3...)
  centroid_embedding vector(768),  -- Representative embedding for the cluster
  representative_error TEXT,  -- Most representative error message
  test_count INTEGER NOT NULL DEFAULT 1,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),

  UNIQUE(execution_id, cluster_id)
);

-- Index for fast cluster lookup
CREATE INDEX IF NOT EXISTS idx_failure_clusters_execution
ON failure_clusters(execution_id);

-- Link table: which test_results belong to which cluster
CREATE TABLE IF NOT EXISTS failure_cluster_members (
  id SERIAL PRIMARY KEY,
  cluster_id INTEGER NOT NULL REFERENCES failure_clusters(id) ON DELETE CASCADE,
  test_result_id INTEGER NOT NULL REFERENCES test_results(id) ON DELETE CASCADE,
  distance_to_centroid FLOAT,  -- How close to cluster center

  UNIQUE(test_result_id)  -- Each result belongs to exactly one cluster
);

CREATE INDEX IF NOT EXISTS idx_cluster_members_cluster
ON failure_cluster_members(cluster_id);

CREATE INDEX IF NOT EXISTS idx_cluster_members_result
ON failure_cluster_members(test_result_id);

-- Add comments
COMMENT ON TABLE failure_clusters IS
  'Pre-computed failure clusters for fast dashboard loading. Each cluster groups semantically similar failures.';

COMMENT ON TABLE failure_cluster_members IS
  'Maps test_results to their assigned failure cluster.';
```

**Design Decision:**
- Caching clusters enables instant dashboard loading
- `centroid_embedding` allows finding the "most representative" failure
- `distance_to_centroid` helps identify outliers
- Foreign keys with CASCADE ensure cleanup

**Verification:**
- [ ] Both tables created with proper relationships
- [ ] Indexes for query patterns
- [ ] Comments explain purpose
